## 1. CONTEXTO DO PROJETO

### O que é o FamilyFlow
Assistente Inteligente Familiar que reduz a carga mental das famílias através de IA, organização compartilhada e planejamento inteligente. Não é um app de tarefas genérico — é um centro de comando familiar com IA proativa.

### Stack Tecnológica
- **Frontend:** React 19 + TypeScript + Vite + Tailwind v4
- **Backend:** Express + TypeScript (serverless no Vercel)
- **Estado:** Zustand (stores por domínio)
- **Roteamento:** React Router v7 (lazy loading)
- **Banco:** Firestore (collections normalizadas)
- **IA:** Google Gemini API (@google/genai)
- **Tempo Real:** Socket.IO (polling-only em produção/Vercel)
- **Testes:** Vitest (121 testes passando)
- **Ícones:** Lucide React
- **Animações:** Motion (motion)

### Deploy
- **Frontend + API:** Vercel (serverless function em `api/index.ts`)
- **URL:** https://familyflow-lake.vercel.app
- **Repositório:** https://github.com/Blackfml/Familyflow

## 4. ARQUITETURA — MELHORIAS ESTRUTURAIS

### 4.1 Monorepo (Opcional, alta complexidade)
Separar em pacotes para melhor organização:
```
familyflow/
├── packages/
│   ├── server/           # Express API (controllers, services, middleware)
│   ├── web/              # React SPA (pages, components, stores)
│   └── shared/           # Types, utils, constants compartilhados
├── docs/
└── .github/
```
- **Benefício:** Separação clara, builds independentes, tipos compartilhados
- **Risco:** Precisa refatorar imports, paths, build scripts

### 4.2 Clean Architecture (Média prioridade)
Implementar camadas corretamente nos controllers/serviços:
- **Domain:** Entidades, interfaces (já existe em `src/types.ts`)
- **Application:** Use cases, orquestração
- **Infrastructure:** Repositories, datasources
- **Presentation:** Pages, components

**Problema atual:** Controllers chamam `stateService.get()` diretamente. Ideal: Controller → Service → Repository → Firestore.

**Arquivos para refatorar:**
- `server/controllers/*.ts` — Remover acesso direto ao stateService
- `server/services/*.ts` — Adicionar lógica de negócio
- `server/repositories/*.ts` — Já existem goal.repository.ts e task.repository.ts, criar para os demais

### 4.3 Cache Layer
Adicionar cache para reduzir leituras no Firestore:
- **Servidor:** Cache em memória com TTL (já existe via stateService, mas não expira)
- **Cliente:** IndexedDB para cache offline
- **Estratégia:** Cache-first com revalidação via WebSocket

### 4.4 Multi-tenant (Futuro)
Preparar arquitetura para múltiplas famílias:
- Coleções já têm `{familyId}` no path
- `stateService` precisa aceitar `familyId`
- Auth middleware precisa extrair `familyId` do JWT

## 10. REGRAS PARA O LOVABLE

### Imperativo — NÃO QUEBRAR
- ❌ Não remover testes existentes
- ❌ Não mudar a estrutura de exportação do Vercel (`api/index.ts` export default app)
- ❌ Não remover `includeFiles: server/**` do vercel.json
- ❌ Não mudar a assinatura dos endpoints da API (frontend depende deles)
- ❌ Não remover tipos existentes em `src/types.ts`
- ❌ Não mudar o nome das stores ou seus métodos públicos
- ❌ Não adicionar dependências pesadas sem verificar bundle size

### Style Guide
- **TypeScript strict mode** (já configurado)
- **Tailwind v4** para estilização (usar classes, não CSS modules)
- **Lucide React** para ícones
- **Motion** para animações (não adicionar framer-motion separado)
- **Portugês brasileiro** em toda UI e mensagens
- **Async/await** para operações assíncronas (não .then/.catch)
- **Zustand** para estado global (não Redux, não Context API)
- **Vitest** para testes (não Jest, não Mocha)

### Commits
Usar conventional commits:
```
feat: adiciona onboarding flow
fix: corrige validação de email no registro
refactor: extrai lógica de cache para serviço separado
perf: reduz bundle size com dynamic imports
a11y: adiciona ARIA labels nos botões
pwa: implementa service worker com cache
```
